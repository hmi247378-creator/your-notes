# 你的笔记（Your Notes）架构设计说明

> 目标：用“统一后端 + 多端前端 + 可插拔分类引擎”的方式，先快速交付 Phase 1 MVP（6 周），并为 Phase 2 智能升级预留接口与数据闭环。

## 一、范围与目标

### 1.1 本文覆盖范围
- **PC Web 端**：主操作平台（标签树管理、分屏编辑、搜索、批量管理、导出、统计面板）。
- **微信小程序**：极速记录与查看（首页即输入框、常用标签、离线存储、弱网同步）。
- **后端服务**：用户、标签、笔记、搜索、同步、分类、统计、导出、提醒（后续）。
- **智能分类**：Phase 1 关键词/规则/统计匹配；Phase 2 向量语义 + 个性化重排。

### 1.2 关键指标（来自需求文档）
- **页面加载**：< 2 秒
- **分类响应**：< 1 秒
- **同步延迟**：< 5 秒
- **服务端响应**：< 200 ms（常规读写接口目标）
- **并发**：1000 在线用户（MVP 以“可水平扩容”为目标）
- **小程序包体积**：主包 ≤ 2MB

## 二、总体架构

### 2.1 架构结论
- **模块化单体优先**：先做一个 `api-service`（逻辑模块清晰拆分），避免早期微服务的复杂度。
- **可插拔分类引擎**：分类能力通过统一接口抽象；Phase 1 与 Phase 2 可平滑替换。
- **多端统一数据协议**：Web 与小程序共用 API 与 DTO/Schema（便于联调与减少返工）。

### 2.2 逻辑模块
- **Auth 模块**：注册/登录/会话与鉴权（Phase 1 支持账号密码或简化登录；微信登录可占位）。
- **Tags 模块**：标签树 CRUD、移动、合并、颜色标记、树查询。
- **Notes 模块**：笔记 CRUD、多标签关联、归档/删除、批量操作。
- **Search 模块**：全文检索（Phase 1 Postgres FTS）。
- **Classify 模块**：输入文本 -> 返回 top3 建议 + 置信度；记录建议与反馈。
- **Sync 模块**：变更日志 + 增量拉取 + 推送（离线补偿、幂等）。
- **Analytics 模块**：分类效果与使用统计（Phase 1 先打通埋点/指标入库）。
- **Export 模块**：导出 Markdown/PDF/Excel（Phase 1 可仅 Markdown，PDF/Excel 走 Phase 2）。
- **Reminder 模块**：标签定时提醒、每日回顾（Phase 2+）。

### 2.3 运行时组件（推荐）
- **API 服务**：NestJS（TypeScript）
- **数据库**：PostgreSQL（主库）
- **缓存/队列**：Redis + BullMQ（异步任务：导出/统计/画像更新）
- **对象存储**：S3/MinIO（图片、导出文件、OCR 结果缓存）

## 三、数据建模（核心）

### 3.1 标签树实现方式
推荐使用 **Materialized Path（可选 ltree）**：
- **优势**：子树查询快、树展示简单、移动可控，适合“按标签筛选/统计/批量操作”。
- **限制**：标签深度建议限制 **≤ 7**（需求建议）。

标签关键字段：
- `parentId`：父标签
- `path`：如 `work.projectA.meeting`（或 ltree 类型）
- `depth`：层级深度（用于校验 ≤ 7）

### 3.2 核心实体关系
- 用户 → 多个标签树（同一用户一棵树也可，后续可支持多棵）
- 标签 → 多个子标签（树结构）
- 笔记 → 多个标签（多对多）
- 分类建议/反馈 → 用于优化分类模型（闭环数据资产）

### 3.3 同步策略（离线友好）
采用 **变更日志（Change Log）+ 幂等推送**：
- 小程序离线时将“变更操作”写入本地队列 `pendingChanges[]`
- 联网后批量 `sync/push`，服务端写入数据并记录 `change_log`
- 客户端使用 `sync/pull?since=...` 增量拉取变更（包含自己的与跨端产生的变更）

幂等关键：
- 每个变更携带 `clientMutationId`（UUID），服务端记录已处理集合，避免重复提交产生多次写入。

## 四、智能分类（可插拔）

### 4.1 分类接口统一抽象
分类引擎必须满足：
- **同步返回**（<1s）
- **top3 建议** + `score` + `level`（high/mid/low）
- **可选解释**（Phase 1 更容易解释，利于用户信任）

分类引擎接口（概念）：
- 输入：`inputText`, `userId`
- 输出：`suggestions[3]`（tagId/score/level）+ explain（可选）

### 4.2 Phase 1（MVP）策略
- 文本预处理：基础清洗 + 分词（中文）+ 去停用词
- 标签画像（MVP）：
  - 用户手动配置关键词（可选）
  - 系统从该标签下最近 N 条笔记抽取高频词（异步更新）
- 打分：TF-IDF/BM25 + 规则加权（例如“会议/纪要/项目”等词）
- 置信度：基于 top1-top2 分差、绝对分数区间映射为 high/mid/low

### 4.3 Phase 2（智能核心）策略
- 向量召回：embedding + `pgvector` topK
- 重排：结合规则、近期偏好（最近常用标签）、时间与上下文
- 个性化：利用 `classification_feedback` 做用户偏好增强

## 五、端侧架构要点

### 5.1 PC Web（主操作）
- **左侧标签树**：展示/创建/移动/合并/删除
- **右侧内容区**：列表/详情/编辑（MVP 先 Markdown）
- **入口录入**：全局可呼出的输入框（提交即分类建议）
- **批量操作**：批量改标签/归档/删除（MVP 至少支持前两项）

### 5.2 小程序（极速 + 离线）
- **首页即输入框**：减少路径
- **常用标签**：最近使用/置顶（MVP 做最近使用）
- **离线队列**：无网可创建笔记与标签调整（标签管理重功能仍建议在 PC）
- **同步状态**：离线/同步中/失败可重试

包体积策略：
- 小程序主包只放“输入、列表、详情、同步”
- 语音/OCR/统计/导出等放分包或后置到 Phase 2

## 六、非功能性：安全、备份、可观测

### 6.1 安全
- HTTPS、鉴权、最小权限
- 数据加密：优先“服务端静态加密（at-rest）”
- **端到端加密（可选）**：会显著影响搜索与分类（需客户端索引/加解密），建议 Phase 3 作为高级功能评估

### 6.2 备份与恢复
- 每日自动备份 Postgres（MVP：cron + 备份文件）
- 支持手动备份（触发导出/备份任务）

### 6.3 可观测
- 关键接口耗时（classify、sync、search）
- 分类命中率/修正率
- 同步成功率/失败率与重试次数
